#+TITLE: CMake 深度进阶教程

本仓库是一个可直接构建和运行的现代 CMake 教程示例. 它通过一个可执行文件和四个内部库, 演示目标导向的依赖建模, 而不是手工维护编译和链接参数.

如果你想先看英文版入口说明, 请参考 [[file:README.md][README.md]].

* 这个仓库演示什么

- =libA=: 静态库, 演示 =PRIVATE= 的实现边界
- =libB=: 静态库, 通过 =PUBLIC= 链接 =libA=, 演示依赖传播
- =libC=: 头文件库, 演示 =INTERFACE= 目标
- =MathFunctions=: 演示配置阶段的符号探测与条件链接
- =Tutorial=: 汇总所有模块输出, 展示最终效果

* 环境要求

- CMake 3.15 或更新版本
- 支持 C++14 的 C++ 编译器
- 可用的 C 编译器, 因为配置阶段会通过 =CheckSymbolExists= 检查数学函数
- 支持运行生成构建工具的 shell 环境

本工作区已在 2026-04-11 使用 macOS + AppleClang 21 验证过配置, 构建, 测试, 安装和运行流程.

* 快速开始

始终使用 *源外构建*:

#+begin_src bash
cmake -S . -B build
#+end_src

编译整个项目:

#+begin_src bash
cmake --build build
#+end_src

运行测试:

#+begin_src bash
ctest --test-dir build --output-on-failure
#+end_src

运行示例程序:

#+begin_src bash
./build/src/Tutorial 36
#+end_src

安装到本地 staging 目录:

#+begin_src bash
cmake --install build --prefix build/install
#+end_src

运行安装后的程序:

#+begin_src bash
./build/install/bin/Tutorial 36
#+end_src

* 运行结果应该包含什么

程序会按章节输出以下内容:

- 项目名和版本信息
- =libA= 的 =PRIVATE= 示例输出
- =libB= 的 =PUBLIC= 依赖传播输出
- =libC= 的头文件库输出
- =MathFunctions= 的平台能力探测结果, 例如 =LOG=, =EXP=, =SQRT=

* 目录说明

- =CMakeLists.txt=: 顶层项目声明, 配置头生成, 测试和安装规则
- =src/=: =Tutorial= 可执行程序, 负责串联所有教学输出
- =libA/=: 演示 =PRIVATE= 编译选项和公共头文件
- =libB/=: 演示 =PUBLIC= 链接如何把 =libA= 传播给消费者
- =libC/=: 演示头文件库的 =INTERFACE= 用法
- =MathFunctions/=: 演示配置阶段能力探测以及按需链接数学库
- =CMake-Best-Practices---2nd-Edition-main/=: 上游参考资料, 不属于根目录教程主流程

* 建议阅读顺序

如果你是为了学习项目结构, 建议按这个顺序阅读:

1. =CMakeLists.txt=
2. =libA/CMakeLists.txt=
3. =libB/CMakeLists.txt=
4. =libC/CMakeLists.txt=
5. =MathFunctions/CMakeLists.txt=
6. =src/CMakeLists.txt=
7. =src/main.cpp=

这种顺序先建立目标图, 再看最终可执行文件如何消费这些目标, 更容易理解现代 CMake 的依赖传播模型.

* 概念与目标映射

| 目标           | 类型      | 教学重点                           |
|----------------+-----------+------------------------------------|
| =libA=         | 静态库    | =PRIVATE= 编译选项与实现边界       |
| =libB=         | 静态库    | =PUBLIC= 链接依赖传播              |
| =libC=         | 接口库    | 头文件库与 =INTERFACE= 使用需求    |
| =MathFunctions=| 静态库    | 配置阶段能力探测与条件平台链接     |
| =Tutorial=     | 可执行文件| 通过声明依赖消费目标, 而不是手排链接顺序 |

* 构建, 测试, 安装要点

- 先配置, 再构建, 再测试或安装
- 仅执行配置还不够, =ctest= 需要先生成 =build/src/Tutorial=
- =cmake --install= 同样依赖已构建的库和可执行文件
- 默认未安装的可执行文件路径是 =build/src/Tutorial=
- 使用本文命令安装后, 本地安装前缀位于 =build/install/=

* 故障排查

** =ctest= 提示找不到 =build/src/Tutorial=

这是因为测试在构建完成前执行了. 请先运行:

#+begin_src bash
cmake --build build
ctest --test-dir build --output-on-failure
#+end_src

** =cmake --install= 报错找不到静态库或可执行文件

安装必须在成功构建之后执行:

#+begin_src bash
cmake --build build
cmake --install build --prefix build/install
#+end_src

** 配置阶段显示 =MathFunctions: Linking against math library 'm'=

这通常不是错误, 而是平台需要显式链接数学库时的正常探测结果.

** 切换生成器, 编译器或 toolchain 之后配置异常

删除旧的构建目录后重新配置:

#+begin_src bash
rm -rf build
cmake -S . -B build
#+end_src

** 不清楚程序应该从哪里运行

- 未安装前运行 =./build/src/Tutorial=
- 使用本文示例前缀安装后运行 =./build/install/bin/Tutorial=

* 延伸说明

现代 CMake 的核心不是“写一堆命令”, 而是“声明目标的使用需求”. 一旦目标之间的关系清晰, 头文件路径, 编译特性, 链接依赖和部分平台差异都可以通过目标图自动传播.

本仓库里最值得观察的点有三个:

1. =libB= 通过 =PUBLIC= 链接 =libA=, 所以 =Tutorial= 只声明直接依赖即可
2. =libC= 作为 =INTERFACE= 库, 没有源码文件, 但会把头文件目录和编译要求传播给消费者
3. =MathFunctions= 在配置阶段探测 =log=, =exp=, =sqrt=, 体现了“先发现平台能力, 再决定链接方式”的思路
